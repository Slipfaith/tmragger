# Задача: доработка TM Repair

Нужно внести два изменения: детализировать фильтр отчёта и добавить офлайн-воркфлоу для разметки правок.

---

## 1. Детализированный фильтр отчёта

**Проблема:** сейчас в отчёте фильтр `"Все типы"` и отдельный фильтр для Split — всё остальное свалено в одну кучу. Нужно фильтровать по каждому типу проверки отдельно.

**Требования:**

- Замени текущий dropdown / селектор на **мультивыборный** фильтр с чекбоксами по каждому типу проверки:
  - Удаление дублей
  - Очистка пробелов
  - Удаление служебной разметки
  - Удаление мусорных TU
  - Split
  - (и все остальные типы, которые программа реально умеет — пройдись по коду и собери полный список)
- Сверху — чекбокс **"Все типы"**, который переключает все остальные разом (трёхсостояние: все / ни одного / частично).
- Рядом с каждым типом показывай счётчик найденных проблем этого типа — чтобы сразу видеть, где густо.
- Фильтр применяется к таблице отчёта мгновенно, без нажатия "Применить". Состояние фильтра сохраняется между запусками программы.
- Если в проекте уже есть перечисление (enum / список констант) типов проверок — используй его как источник правды, не хардкодь строки в UI.

## 2. Офлайн-воркфлоу: экспорт отчёта → разметка вне программы → импорт решений

**Суть сценария:**
1. Программа проверила TM, показала отчёт с предложенными правками.
2. Я нажимаю **Экспортировать пакет** — получаю один файл `.tmrepair` (ZIP-контейнер) со всем, что нужно для офлайн-разметки.
3. Закрываю программу, иду размечать — открываю пакет, внутри лежит XLSX (или HTML), размечаю accept/reject, сохраняю пакет.
4. Открываю программу, импортирую тот же `.tmrepair` — программа применяет только accepted правки и сохраняет исправленный TMX.

### 2.1 Формат пакета `.tmrepair`

Обычный ZIP-архив с расширением `.tmrepair`. Структура:

```
<name>.tmrepair
├── manifest.json          # метаданные, версия формата, хеши
├── source.tmx             # копия оригинального TMX на момент проверки
├── state.json             # список проблем + статусы (pending/accepted/rejected/skipped)
├── report.xlsx            # редактируемая таблица для пользователя
└── report.html            # альтернативная HTML-форма для разметки в браузере
```

**manifest.json** содержит:
- версию формата пакета (для будущей совместимости)
- имя и хеш (SHA-256) исходного TMX
- настройки проверки, с которыми был сгенерирован отчёт
- временные метки создания и последнего изменения

**state.json** — источник правды для программы:
- стабильные ID проблем (например, `tu_index + issue_type + position`)
- для каждой проблемы: тип, TU source/target, описание, предлагаемая правка, статус (`pending` / `accepted` / `rejected` / `skipped`), комментарий

**report.xlsx и report.html** — производные от state.json, но именно с ними работает пользователь офлайн.

### 2.2 Экспорт пакета

- Кнопка **Экспортировать пакет** → диалог сохранения, default-имя: `<tmx_name>_<timestamp>.tmrepair`.
- Внутри пакета сразу формируются оба файла разметки — пусть пользователь сам решает, чем удобнее размечать:
  - **XLSX:** колонки — ID проблемы, тип проверки, TU source, TU target, описание, предлагаемая правка, **Решение** (выпадающий список: accept / reject / skip), **Комментарий**. Все колонки кроме Решения и Комментария защищены от правки — чтобы случайно не сбить ID.
  - **HTML:** самодостаточный файл с инлайн-JS, без внешних зависимостей. Таблица с теми же колонками, кнопки accept/reject прямо в строках, кнопка **Скачать размеченный файл** сверху — выгружает обратно JSON с решениями.
- Первая строка XLSX / скрытый блок HTML содержит ID пакета и хеш TMX — для привязки при импорте.

### 2.3 Разметка офлайн

Пользователь может:

- Открыть `.tmrepair` в программе-архиваторе, вытащить `report.xlsx`, разметить, положить обратно в архив (заменив файл). Программа при импорте подхватит обновлённый XLSX.
- **Или** открыть `report.html` прямо в браузере, разметить, скачать размеченный JSON, и положить его в корень архива как `decisions.json`. Программа при импорте проверит наличие этого файла в первую очередь.
- **Или** оставить `.tmrepair` нетронутым и указать программе отдельно размеченный XLSX / JSON — тогда он подгружается рядом с оригинальным пакетом.

### 2.4 Импорт пакета

- Кнопка **Импортировать пакет** принимает `.tmrepair`.
- Программа:
  - читает `manifest.json`, `source.tmx`, `state.json`
  - ищет разметку в таком порядке: `decisions.json` → обновлённый `report.xlsx` → `report.html` → внешний файл, указанный пользователем
  - сверяет хеш TMX — если на диске есть более новая версия TMX с тем же именем, показывает предупреждение
  - сверяет ID проблем с `state.json`, отмечает несовпадения (если пользователь вручную добавил строки или удалил — это должно быть видно)
  - обновляет статусы в `state.json` внутри пакета
  - показывает сводку: **N accepted, M rejected, K skipped, L неопознанных**
- После импорта кнопка **Применить к TMX** — программа применяет только `accepted` правки и сохраняет новый TMX (не перетирая оригинал — рядом с суффиксом `.repaired.tmx`).

### 2.5 Восстановление сессии

- При запуске программа сканирует последние открытые папки на наличие `.tmrepair` рядом с открытыми TMX и показывает в списке недавних.
- При открытии `.tmrepair` напрямую (через Open Recent или drag-and-drop) — загружает состояние, отчёт и TMX из пакета, сразу готова к импорту разметки или к продолжению работы.

## Общие требования

- UI фильтров и экспорта/импорта — нативно PySide6, не HTML-виджеты внутри Qt.
- Никакой потери данных при сбое: все операции с пакетом — через временный файл и атомарное переименование.
- Пакет самодостаточен — его можно переслать коллеге, и он откроет его у себя без дополнительных файлов.
- Стиль кода и структуру модулей согласовывай с существующим проектом.
- Перед правками покажи план файлов, которые будешь менять, и жди подтверждения, если изменений больше 3 файлов.

## Критерии приёмки

1. В отчёте можно включить только "Удаление дублей" и "Очистка пробелов" — остальные типы скрыты, счётчики сверху отражают выбор.
2. Сценарий end-to-end: запустить проверку → экспорт `.tmrepair` → закрыть программу → разметить XLSX внутри архива → открыть программу → импорт `.tmrepair` → применены только accepted правки, получен `<name>.repaired.tmx`.
3. Тот же сценарий с HTML-разметкой и `decisions.json` работает идентично.
4. Пакет, отправленный по почте на другой компьютер, открывается без проблем и содержит всё необходимое для разметки.
5. Изменение TMX между экспортом и импортом корректно детектируется по хешу и показывает предупреждение.
